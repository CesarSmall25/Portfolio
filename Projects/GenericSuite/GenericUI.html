Generic UI

A simple expansion for even easier Common UI support

What is it?

Generic UI is a plugin that expands on top of the Common UI plugin provided by Epic Games for Unreal Engine. It provides additional functionality and features that make it even easier to create and manage user interfaces in Unreal Engine

The core concept is that every UI element is a widget that can be created, shown, hidden and destroyed through simple blueprint nodes.
It also provides a UI Manager that handles the creation and management of these widgets or a navigation component that can be added for easy local navigation. Both are valid options and ultimately have the same functionality under the hood, with a few extra steps through the manager.

How to use it

Setting up the layers for the UI

Generic UI relies on the Common UI plugin's layer system to manage the different UI elements. Once Common UI is enabled in the project settings, the user needs to create a few things in order to get the system working:

  • A GAppShell class : this is so that we can manage the layers for this player's screen
  • A "Layer" GameplayTag : this is so that we can distinguish the layers from one another. One tag is required per type of stack we want on-screen
Once the app shell is created, the user needs to set GenericWidgetStacks for each layer they wish to use in the app shell. This can be as many layers as the user wants but I usually make sure I have at least one for the HUD, one for the pause menus and one for modal popups and one for toasts or notifications.

Inside of each WidgetStack the user needs to set the GameplayTag created earlier as the Layer Tag for that stack. This is how the system knows which stack to push or pop widgets from.

Using the subsystem

In order to use the subsystem the user simply needs to get a reference to it through the "Get GNavigationService Subsystem" node. From there the user can call Push or Pop with a reference to the requested playercontroller and push ot pop to the desired widget stack. app shell is correctly set up the user now has the option on how they want to instantiate the app shell. They can either set it up using the GHUD class which has a property for the appsghell and will build it once the player's playercontroller is ready or can implement their own logic to create it through blueprints or C++. If using the GHUD class, once everything is set the HUD will call an OnHUDCreated() function flagging that the app shell is ready to be used.

Using the component

Similarly to the subsystem and HUD method, the user can also add the Navigation Component to their player controller and define the appshell they wish to use. From there it's the same as before, calling Push and Pop on the component reference.
